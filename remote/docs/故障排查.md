# 故障排查

## `ssh` 可以登录，但脚本提示找不到 `artiq_run`

原因通常是：交互式登录会加载 `.bashrc`，但远程非交互命令没有加载你的 ARTIQ 环境。

解决方法：在 `config.local.json` 设置 `remote_env_setup`。

conda 示例：

```json
"remote_env_setup": "source ~/miniconda3/etc/profile.d/conda.sh && conda activate artiq"
```

虚拟环境示例：

```json
"remote_env_setup": "source ~/artiq-venv/bin/activate"
```

然后测试：

```powershell
python remote_artiq.py check --config config.local.json --deep
```

## `scp` 上传失败

常见原因：

- 工控机用户名或 IP 错误。
- `remote_project_dir` 没有权限写入。
- 工控机磁盘满了。
- SSH 端口不是 22，但配置里没有改。

建议先手动测试：

```powershell
ssh 用户名@工控机IP "mkdir -p ~/artiq_remote_jobs && echo ok"
```

## `artiq_run` 报 core device 连接错误

这说明当前电脑已经成功让工控机执行命令，但工控机和 ARTIQ core device 之间有问题。

检查：

- 工控机能否 `ping` 到 core device。
- `device_db.py` 里的 `core_addr` 是否正确。
- core device 是否上电、网口灯是否正常。
- 工控机是否选择了正确的网卡和网段。

## `ttl_blink.py` 报找不到 `ttl0`

`ttl0` 只是示例设备名。你的系统里可能叫 `ttl4`、`ttl_out0`、`urukul0_ch0_sw` 等。

在工控机上打开 `device_db.py`，查找实际 TTL 设备名，然后修改 `examples/ttl_blink.py`：

```python
self.setattr_device("ttl0")
```

把 `ttl0` 改成实际名称。

## `artiq_client` 提交失败

先确认你是否真的需要 `artiq_client` 方案。最简单的 SSH/SCP 方案不需要本机安装 ARTIQ。

如果使用 `artiq_client`：

- 当前电脑需要安装 ARTIQ 客户端。
- 客户端版本最好与工控机一致。
- 工控机需要先启动 `artiq_master`。
- 工控机防火墙需要放行 3250 和 3251。

检查端口：

```powershell
python remote_artiq.py check --config config.local.json
```

## 中文路径和空格路径

本项目放在 `Z:\Wang Junjie\Artiq_Project\remote`，这个路径里有空格。脚本已经按普通 Windows 路径处理。

但是工控机上的 `remote_project_dir` 建议不要包含中文或空格，保持类似：

```json
"remote_project_dir": "~/artiq_remote_jobs"
```

这样可以避免 Linux shell、SSH、SCP 对特殊字符处理不一致。
